### Issue 1 — Kernel để ở `cms/` hay `src/cms/`?

Architecture doc nói rõ: kernel (meta.js, markdown.js, config.js...) phải đặt ở `cms/` ngoài root — tách khỏi `src/` hoàn toàn. Lý do là để enforce isomorphic boundary cứng: code trong `cms/` không được gọi bất kỳ thứ gì của environment (không `fs`, không `window`, không `process`). Structure thay cho discipline.

Nhưng test fixtures trong 04878aa lại import từ `../meta.js` — tức là chúng expect implementation nằm ở `src/cms/`. Còn `cms/` ngoài root thì trống hoàn toàn.

Hai hướng:

**Option A — Giữ `cms/` ngoài root (theo doc):** Đúng với intent ban đầu, ADR-001 intact, boundary rõ ràng bằng cấu trúc thư mục. Cần update import trong test fixtures.

**Option B — Dùng `src/cms/` (theo tests):** Familiar hơn với JS convention thông thường, dev mới onboard nhanh hơn. Nhưng phải update architecture doc và ADR-001. Boundary giữa kernel và platform code từ đây sẽ phụ thuộc vào discipline của team, không còn enforce bằng structure nữa.

**Decision:** [x] Option A — Giữ `cms/` ngoài root, update test imports.

### Issue 2 — Config dùng YAML hay chuyển sang JSON?

Doc quy định config CMS là `cms/config.yaml`, fixtures cũng là `.yaml`. Nhưng `FS.load()` chỉ auto-parse `.json` — YAML trả về raw string. `YAML.js` hiện tại chỉ có `stringify` để ghi, không có `parse` để đọc. Kết quả: `loadConfig()` hiện tại không chạy được.

Điều này hơi mâu thuẫn vì anh Huy đã giải thích rõ trong doc tại sao article metadata dùng JSON thay vì YAML: YAML phức tạp, edge cases nhiều, zero-dep nên không kéo thư viện vào. Đúng hết. Nhưng config lại đang dùng YAML.

Ba hướng:

**Option A — Đổi config sang JSON:** Consistent với lý do đã document. Sửa fixtures, không cần viết parser. Xong.

**Option B — Viết YAML parser trong `YAML.js`:** Giữ config dạng YAML (dễ đọc hơn, AI viết config cũng dễ hơn). Phải implement parser cho subset YAML đủ dùng: flat/nested objects, string, number, boolean, array. Rủi ro là YAML có rất nhiều edge cases — viết parser đủ robust tốn thời gian và vẫn dễ bug.

**Option C — Config là JS module (`config.js`):** Import trực tiếp, không cần parse gì, validate được bằng code. Đánh đổi: non-developer không tự sửa được nữa.

**Decision:** [x] Option A — Đổi config sang JSON, sửa fixtures.

### Issue 4 — Bugs trong IDB và States, cần fix trước khi đi tiếp

Review kỹ thấy một số bugs sẽ gây đau đầu nếu để nguyên:

- **`IDB.js` — `new Promise(async executor)`:** Exception bên trong async executor bị swallow. Promise không bao giờ resolve hoặc reject nếu throw bên trong — fail silent, khó debug.

- **`IDB.js` + `disk.js` — import name mismatch:** Field `load`/`save` có thể là `undefined` nếu import name không khớp. Gọi `put` trên Node sẽ crash ngay.

- **`IDB/put.js` — array key coerce thành string trên browser:** Browser path truyền array làm IDB key, bị coerce thành `"a,b"`. Không round-trip được với Node path — data lưu xuống nhưng đọc lại không ra.

- **`States/set.js` — re-entrant notification drain:** Nếu subscriber gọi `set()` lại bên trong callback thì notifications bị drop hoặc ra sai thứ tự.

- **`States/on.js` — silent `undefined`:** Return `undefined` thay vì throw khi key được pass nhưng không có subscriber. Callsite crash sau đó theo kiểu khó trace.

**Action:** Fix hết trước khi build bất kỳ layer nào dùng IDB hoặc States.
