Hôm nay muốn hiểu rõ phần AdSense và xem có thể nhảy thẳng vào implement nó không, thay vì đi theo thứ tự kernel → builder → UI từ đầu.

---

### Kết luận: phải theo flow — vì flow đó chính là con đường đến AdSense

AdSense trên static site không phải một module riêng biệt. Nó là **output requirement** — yêu cầu đặt ra cho toàn bộ build pipeline:

```
config.js ──┐
meta.js    ─┤→ render.js → index.html (có AdSense script + 3 ad slots)
markdown.js ┤
seo.js     ─┘
```

Phần AdSense thực sự chỉ gồm 2 dòng code trong `render.js`:

1. `<script async src="//pagead2.googlesyndication.com/pagead/js/adsbygoogle.js" crossorigin="anonymous"></script>` trong `<head>` của output HTML.
2. Ba `<div>` placeholder: `ad-top`, `ad-mid`, `ad-bottom` ở đúng vị trí trong body.

Nhưng để viết `render.js`, cần có:
- `config.js` — biết `site.url`, `default_og_image`, publisher config
- `meta.js` — biết title/description/slug của bài
- `markdown.js` — có HTML body của bài để đặt ad slot vào giữa
- `seo.js` — có OG tags (thiếu OG image → AdSense review có thể reject)

Không thể implement `render.js` với output đúng nếu inputs của nó chưa tồn tại. Nhảy thẳng vào AdSense = xây nhà từ tầng 2.

---

### AdSense yêu cầu gì từ output HTML — ghi để nhớ khi implement render.js

| Yêu cầu | File chịu trách nhiệm | Ghi chú |
|---|---|---|
| Pre-rendered HTML (không phải JS render) | `render.js` | Crawler không chạy JS — HTML phải có content ngay |
| `<script async>` AdSense trong `<head>` | `render.js` | Phải có `async`, không được `defer` cho AdSense script này |
| 3 ad slot `<div>` trong body | `render.js` + `cms-page` | `ad-top` trước nội dung, `ad-mid` giữa, `ad-bottom` sau |
| Không có `attachShadow` | `cms-page/index.js`, `cms-list/index.js` | Shadow DOM = crawler không đọc được ad slot |
| OG tags đầy đủ (`og:image`, `og:title`, `og:description`) | `seo.js` | Thiếu `og:image` → AdSense review reject |
| Không có blocking script nào khác | `render.js` (hard gate) | Mọi `<script>` khác phải có `defer`/`async`/`type="module"` |

## Research cần làm trước khi viết render.js và seo.js

### 3 câu hỏi sống còn (John — PM)

**1. Google AdSense crawler đọc HTML như thế nào?**
- Googlebot có chờ `DOMContentLoaded` không, hay chỉ đọc raw HTML từ server?
- `<ins class="adsbygoogle">` phải nằm ở đâu để crawler ghi nhận content context tốt nhất?
- Nếu ad slot nằm trong Web Component (light DOM, không Shadow DOM), crawler có bỏ qua không?

**2. AI-generated content + AdSense: penalize như thế nào?**
- 50–900 bài/ngày là con số aggressive — ngưỡng nào bị flag manual review?
- Site structure (URL, breadcrumb, internal link density) có ảnh hưởng đến Google gom nhóm "thin content" không?
- Phải biết trước khi design `seo.js` vì meta tags là tuyến phòng thủ đầu tiên

**3. Pre-rendered static HTML + 18 locales: hreflang implement như thế nào?**
- `hreflang` sai = traffic organic không tăng dù content tốt
- 18 locales × 900 bài/ngày = sitemap giới hạn 50,000 URL/file, cần sitemap index
- `canonical` vs `hreflang` — không thể dùng cả hai sai cách

**Risks chưa được address (theo thứ tự nguy hiểm):**
1. AdSense approval risk — AI content mỏng → reject + cooldown 30 ngày
2. Core Web Vitals CLS — ad slots không fixed size gây layout shift → penalize ranking
3. Build bottleneck — 16,200 files/ngày, mỗi bài 100ms = 27 phút build
4. `og:locale` alternate tags — ảnh hưởng CTR từ social

---

### Technical gotchas — trap hay dính (Winston — Architect)

| Trap | Vấn đề | File chịu trách nhiệm |
|---|---|---|
| AdSense script `async`-only | Chỉ được `async`, không `defer`, không `type="module"` — conflict với hard gate | `render.js` xử lý như exception |
| `push({})` inline script | Script inline không thể có `defer`/`type="module"` — second conflict | `render.js` documented exception |
| Shadow DOM | Crawler và ad serving không traverse shadow root, CPM drops | `cms-page`, `cms-list` — hard gate đã có |
| CSP headers | AdSense whitelist dài, Netlify/Vercel cần pre-configure | Deploy config |
| Mobile viewport | Thiếu `<meta name="viewport">` → responsive ads không resize → -30–40% revenue | `render.js` template |
| `hreflang` 18 locales | Sai là lỗi phổ biến nhất, ảnh hưởng trực tiếp traffic per market | `seo.js`, `feed.js` |
| IDB/FS cache + ad slots | Cache-First cho HTML có ad slots là sai — ads cần fresh load để count impression | `pipeline.js` cache strategy |
| `canonical` URL | Phải absolute URL, không relative — `routes-inject.js` phải emit đúng format | `render.js`, `seo.js` |

**Research priority order trước khi code:**
1. AdSense script placement rules (async-only + push inline exception)
2. `hreflang` + `canonical` cho 18 locales
3. CSP configuration theo platform deploy
4. Sitemap index format cho >50K URLs
5. Cache strategy — không Cache-First cho HTML có ad slots

**Nguồn đọc trực tiếp (không qua secondary):**
- Google AdSense Publisher Policies → "Ad behavior" + "Valuable inventory"
- Google Search Central → "hreflang" + "Helpful Content System"

## BMAD Workflow — Hướng dẫn thực chiến

### Nguyên tắc cốt lõi

> **Một mình** khi đã biết mình cần gì. **Party mode** khi cần nhiều góc nhìn trước khi quyết định.

### Bản đồ Agent → Nhiệm vụ

| Agent | Khi nào gọi |
|---|---|
| John (PM) | Viết PRD, define feature scope, prioritize backlog |
| Winston (Architect) | System design, tech stack decision, API contract |
| Amelia (Dev) | Implement story, code review, debugging |
| Mary (BA) | Phân tích yêu cầu mơ hồ, viết acceptance criteria |
| Sally (UX) | User flow, wireframe logic, interaction design |
| Paige (TW) | Docs, diary, spec, changelog, ADR |

### Khi nào dùng cái gì?

```
Một agent đơn              Party Mode
─────────────              ──────────
✓ Task rõ ràng             ✓ Quyết định quan trọng không thể đảo ngược
✓ Implement story cụ thể   ✓ Tranh luận trade-off
✓ Fix bug đã locate được   ✓ Bắt đầu epic mới chưa rõ shape
✓ Viết/edit doc            ✓ Kiến trúc mới cần nhiều góc nhìn
✓ Quick research           ✓ Khi cảm thấy "mình đang thiếu gì đó"
```

### Flow thực tế theo giai đoạn

| Giai đoạn | Lệnh | Output mong đợi |
|---|---|---|
| Khởi động epic mới | `/bmad-party-mode` (John + Winston + Mary) | 3 góc nhìn độc lập, anh tổng hợp quyết định |
| Thiết kế kỹ thuật | `/bmad-agent-architect` → Winston | Architecture decision + ADR draft |
| Break down stories | `/bmad-create-epics-and-stories` | Epics + stories cho sprint |
| Implement story | `/bmad-dev-story` → Amelia | Code + design decision giải thích |
| Review trước merge | `/bmad-code-review` hoặc party mode | Correctness + edge cases |
| Debug bị kẹt | `/gitnexus-debugging` hoặc party mode (Amelia + Winston) | Root cause |
| Viết doc/ADR | `/bmad-agent-tech-writer` → Paige | Structured doc |

**Template A — Party mode mở đầu epic:**
```
/bmad-party-mode

Context: Akao CMS, team 2 devs.
Topic: [Tên feature/decision]

John: Feature này solve được user problem gì? MVP là gì?
Winston: Technical approach và risk chính?
Mary: Yêu cầu nào chưa rõ, cần clarify với ai?
```

**Template B — Single agent implement:**
```
/bmad-dev-story

Story: [title]
File liên quan: [paths]
Acceptance criteria:
- [ ] criterion 1
- [ ] criterion 2
```

**Template C — Research kỹ thuật:**
```
/bmad-party-mode

Context: Akao CMS cần [X] vì [Y].
Câu hỏi: trade-off giữa [option A] vs [option B]?
Winston: technical trade-offs?
Mary: risk và edge cases?
Output format: comparison table + recommendation.
```

**Template D — Chaining agents qua nhiều turn:**
```
Turn 1: Winston (architect) → design
Turn 2: Mary (BA) → acceptance criteria
Turn 3: Amelia (dev) → implement
Turn 4: /bmad-code-review → review
Turn 5: Paige (TW) → doc/ADR
```

### Quy tắc chaining

1. Luôn paste context từ turn trước — agent không nhớ lịch sử
2. Một agent, một output rõ ràng — không hỏi Winston viết code
3. Party mode = anh ra quyết định cuối, không phải agents
4. Paste key decision vào diary ngay sau session, không để quên
